# MCS Conversation Viewer — Project Knowledge

> This document is for Copilot/developer onboarding. NOT committed to git.
> Location: project root (gitignored)

## What This Is

A **Power Apps Code App** (React + TypeScript + Vite) that lets makers and admins review Copilot Studio conversation transcripts. It reads from the `conversationtranscript` Dataverse table and presents a debug-friendly view of what happened in each conversation.

## Tech Stack

- **React 19** + **TypeScript 5.9** + **Vite 7**
- **@microsoft/power-apps** SDK for Dataverse data access
- **@microsoft/power-apps-vite** plugin for local dev (Local Play URL)
- **Vitest** for testing (79 tests — 55 parser + 24 component)
- **react-markdown** + **remark-gfm** for message rendering
- Plain CSS with CSS variables (dark mode support — no UI framework)
- GitHub Actions for release automation, Dependabot, CodeQL

### Data Flow

```
Dataverse (conversationtranscript table)
  → useTranscripts.ts (OData fetch + pagination)
    → raw JSON record
      → parseTranscript() [parseTranscript.ts]
        ├── classifyTranscriptType() → "chat" | "autonomous" | "evaluation" | "design"
        ├── classifyActivity() + parseActivity() → ParsedActivity[] for each raw activity
        ├── mergePlanSteps() → PlanStep[] (merges Triggered+BindUpdate+Finished by stepId)
        ├── extractAdvancedEvents() [advancedEvents.ts] → AdvancedEvent[]
        └── → ParsedTranscript (final structured object)
          → Components:
              TranscriptList (filters, type badges, infinite scroll)
              TranscriptDetail → GeneralInfo + DebugPanel + MessageTimeline
              AnalyticsSummary (aggregate stats)
```

### Adding a New Pattern (Recipe)

When you discover a new trace valueType, event name, or activity type:

1. **Update `TRANSCRIPT-PATTERNS.md`** — add to master checklist + add JSON snippet
2. **Update `types/transcript.ts`** — add new type/field if needed (e.g. new `ParsedActivityType` value, new field on `ParsedTranscript`)
3. **Update parser** — depends on what it is:
   - New **trace valueType** → add handler in `advancedEvents.ts` (for Advanced debug) or `classifyActivity()` in `parseTranscript.ts` (if it needs dedicated parsing)
   - New **event name** → add case in `classifyActivity()` switch, or let catch-all in `advancedEvents.ts` handle it
   - New **activity type** → add to `RawActivity.type` union, catch-all in `advancedEvents.ts` handles rendering
   - New **channelData key** → update classification logic if it's a signal, or ignore if informational
4. **Add test fixture** in `src/test/fixtures/transcripts.ts` — create a minimal `DataverseTranscriptRecord` with the new pattern
5. **Add test assertions** in `src/test/parseTranscript.test.ts`
6. **Update UI if needed** — usually only for dedicated parsing (new section in DebugPanel, new badge, etc.). Catch-all events render automatically.
7. **Verify** — `npx tsc --noEmit && npx vite build && npx vitest run`

## Transcript Content Structure

The `content` field is a JSON string with `{"activities": [...]}`. Key activity types:

| Type | Name/ValueType | What we extract |
|---|---|---|
| trace | ConversationInfo | Session outcome, locale |
| trace | SessionInfo | Duration, turn count, outcome |
| message (role=0) | — | Bot responses (markdown) |
| message (role=1) | — | User messages |
| event | DynamicPlanStepTriggered | Agent thinking/reasoning |
| event | DynamicPlanStepBindUpdate | Tool arguments |
| event | DynamicPlanStepFinished | Tool results + execution time |
| event | DynamicServerToolsList | Available MCP tools |
| event | UniversalSearchToolTraceData | Knowledge sources searched |
| event | ResponseGeneratorSupportData | Generated response, citations |
| trace | KnowledgeTraceData | Search completion state |
| invoke | message/submitAction (feedback) | Reactions 👍/👎 |
| trace | ErrorTraceData | Errors (advanced mode) |
| event | DynamicServerError | MCP server failures (advanced mode) |
| trace | VariableAssignment | Variable changes (advanced mode, filtered: Topic.CurrentTime excluded) |
| trace | DialogRedirect | Topic switches (advanced mode) |
| trace | EscalationRequested / HandOff | Escalation events (advanced mode) |
| event | DynamicPlanStepBlocked | Content filtering / RAI (advanced mode) |
| trace | IntentRecognition | Intent classification + confidence score (advanced mode) |
| trace | NodeTraceData | Dialog node execution path (advanced mode) |
| message | (with attachments) | Adaptive cards, OAuth cards |
| * | endOfConversation, pvaSetContext, etc. | Catch-all activity rendering (advanced mode) |

## Important Design Decisions

### No $skip — Dataverse doesn't support it
Pagination uses `maxPageSize` + `skipToken`, NOT `skip` (causes error `0x80060888`).

### Bot ID mismatch
The `_bot_conversationtranscriptid_value` lookup field does NOT match `botid` from the bots table. We match on `metadata.BotName` (schema name) instead for agent filtering.

### SharePoint content redaction
Copilot Studio strips bot responses that contain SharePoint knowledge content from the `text` field (security). The response IS available in `ResponseGeneratorSupportData.response`. We show a 🔒 indicator and the knowledge section has the full response.

### Reactions have two formats
- **pva-studio**: `feedback` is an object `{ feedbackText: "..." }`
- **msteams**: `feedback` is a double-encoded JSON string `"{\"feedbackText\":\"...\"}"`

### Orphan reactions
Teams users can react to messages from previous sessions. The `replyToId` points to a message not in the current transcript. We show these in a separate section with on-demand lookup (`contains(content, '<messageId>')` search).

### User identity fallback
If no user messages exist (user opened bot but never typed), the AAD Object ID comes from the `startConversation` event.

### OData injection protection
All user inputs are sanitized via allowlist: `[a-zA-Z0-9\s\-._@:]`. GUIDs are sanitized to `[a-fA-F0-9\-]`.

### Filter state lifted to App.tsx
All list page filter state (date range, search, agent, outcome, feedback, user search query) lives in `App.tsx` as `ListFilterState` so it persists when navigating to detail view and back.

### Multi-Env tab — auth, discovery, access scoping
Cross-tenant transcript browsing without leaving the app. Lives under `src/components/MultiEnv/`.

- **Auth: manual OAuth + PKCE** (NOT MSAL.js). When the app runs in `apps.powerapps.com` as an iframe, Chrome storage partitioning isolates iframe localStorage from the popup's top-level storage, breaking MSAL's cache lookup. We sidestep by running the popup as a self-contained state machine in `src/auth-redirect.ts`: PKCE → AAD authorize → token exchange → broadcast result back.
- **Multi-channel result delivery.** Token is broadcast via three paths simultaneously — `BroadcastChannel`, `window.opener.postMessage`, and `opener.opener.postMessage` — because at least one survives partitioning + COOP from `login.microsoftonline.com`.
- **Per-env tokens via refresh_token.** First sign-in returns `offline_access` refresh token; subsequent env switches exchange it silently for env-scoped access tokens (no re-popup). Cached with 60s pre-expiry refresh in `MultiEnvPanel.tsx`.
- **Env discovery.** `https://api.bap.microsoft.com/providers/Microsoft.BusinessAppPlatform/scopes/admin/environments?api-version=2020-10-01` with management.core.windows.net scope.
- **Access scoping = client-side.** The `conversationtranscript` table is owned by a Microsoft service principal; "Transcript Viewer" gets ALL transcripts in the env regardless of which `/bots` the user can read. We mirror the existing tab's pattern: `$select` the small `metadata` field, parse `BotName` (schema name), intersect with the schema names of bots returned by `/bots` (which IS RLS-scoped). Selected-agent dropdown further narrows by schema name. UI shows "X hidden (not your accessible agents)" so this is visible.
- **App Registration requirements:** redirect URI registered as **Single-page application** (token endpoint CORS); `Dynamics CRM > user_impersonation` delegated permission (admin consent recommended for cross-tenant use).

## Dataverse API Notes (Power Apps Code Apps)

- `ConversationtranscriptsService.getAll()` wraps calls in `$batch` requests internally
- Result shape: `{ success: boolean, data: Record[], skipToken?: string, error?: unknown }`
- The SDK uses `Prefer: odata.maxpagesize=N` for page size control
- Local dev requires the **Local Play URL** printed by `npm run dev` — not just localhost
- The `powerApps()` Vite plugin sets `POWERPLATFORMTOOLS_PACPATH` env var in CI
